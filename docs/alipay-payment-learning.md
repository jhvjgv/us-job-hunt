# 支付宝「网站支付」接口为何这样设计：学习笔记

本文说明开放平台里常见的 **拼跳转 URL + 浏览器访问网关 + 异步通知** 这一套，和「商户服务器直接 HTTP 调支付宝」的差别，以及背后的产品与安全考量。

---

## 1. 你在代码里看到的两条路

| 方式 | 典型形态 | 谁发起对支付宝的请求 |
|------|-----------|----------------------|
| **网站支付（page pay）** | 商户服务端拼好带签名的 URL，前端 `location.href` 跳转 | **用户浏览器** GET `openapi.alipay.com/gateway.do` |
| **异步通知（notify）** | 支付有终态后，支付宝 POST 表单到你的 `notify_url` | **支付宝服务器** → 你的服务器 |

很多教程里说的「调支付宝接口」，在网站支付场景里，**第一步往往是「生成网关请求」而不是服务端发 HTTP**，这是刻意的产品设计，不是遗漏。

---

## 2. 为什么支付发起用「跳转网关」而不是「商户后端直连」？

### 2.1 收银台必须在支付宝域内完成

- 用户需要在 **支付宝托管的页面** 里登录、确认金额、选支付方式（余额/银行卡等）、完成风控与密码/指纹。
- 这些页面和流程在支付宝域名下，**不能**由商户后台一个 API 调用就代替浏览器完成。
- 因此标准做法是：商户只负责 **构造一条符合开放网关规范的请求**（含 `method=alipay.trade.page.pay` 等），让用户 **浏览器** 打开这条 URL，后续交互全部在支付宝侧。

### 2.2 网关 `gateway.do` 的统一入口

- 开放平台大量能力共用 **`https://openapi.alipay.com/gateway.do`**，通过参数 `method` 区分不同产品（支付、查询、退款等）。
- 网站支付走的是 **WAP/PC 页面收银台**，天然适合 **GET 跳转**；签名保证参数未被篡改，支付宝验签后展示对应订单。

### 2.3 和「服务端 SDK 调接口」的关系

- 同一套开放能力，也可以用 **官方 SDK 在服务端** `execute` 调用（例如某些场景需要先创建预下单再调起客户端）。
- **电脑网站支付**文档里推荐的商户侧步骤，常见就是：**组装请求 → 引导用户跳转**。  
- 你的项目用 **手写拼 URL + RSA2 签名**，与 SDK 本质相同，都是生成**支付宝认可的网关请求**，只是没有用 SDK 封装。

---

## 3. 为什么要有异步通知（notify_url）？

### 3.1 同步跳转（return_url）不可靠

- 用户支付完成后，浏览器会跳回你配置的 `return_url`，用于**改善体验**（展示「支付成功」页）。
- 但用户可能关浏览器、断网、浏览器崩溃，**你不能依赖这次跳转来记账**。

### 3.2 以服务端通知为准

- 支付宝在订单进入终态（成功/关闭等）后，向你的 **`notify_url` 发服务端请求**（通常 POST，表单或文档约定格式）。
- **对账、改订单状态、发货、开通会员**等，应以 **验签通过的异步通知**（或主动查单接口）为准，而不是只信 `return_url`。

### 3.3 为什么要返回纯文本 `success`

- 支付宝需要明确知道：**你的系统已收到并处理完这条通知**，否则会按策略**重试**。
- 约定响应体为 **`success`**（且 HTTP 状态成功），是一种简单、可机器判断的握手；返回 JSON 或空 body 可能导致重复通知或告警。

---

## 4. 安全设计：签名与密钥分工

| 密钥 | 典型用途 |
|------|-----------|
| **商户应用私钥** | 你生成请求（拼 URL 或 SDK）时签名，证明请求来自持有该应用的商户 |
| **支付宝公钥** | 你校验支付宝给你的响应、**异步通知**等是否来自支付宝 |

因此：**只拼 URL 不验回调签**，在生产环境是不完整的；**notify 必须验签**后再改数据库。

---

## 5. 两套不同的「支付宝主动通知你」：不要混用

文档中心里会出现多种通知；下面两种最常见、也最容易搞混。

### 5.1 网站 / 手机支付：交易异步通知（`notify_url`）

- **能力**：如 `alipay.trade.page.pay`（电脑网站支付）等在下单时传入的 **`notify_url`**。
- **内容**：表单参数里通常是 **`out_trade_no`、`trade_no`、`trade_status`** 等**交易订单**字段（以该产品的异步通知文档为准）。
- **用途**：用户扫码/网页付商品款后，你据此**改订单、发货、开通会员**。

### 5.2 消息服务：资金单据状态变更（`alipay.fund.trans.order.changed`）

官方文档摘要（以 [资金单据状态变更通知](https://opendocs.alipay.com/) 为准，字段以最新版为准）：

| 项目 | 说明 |
|------|------|
| **适用场景** | 单笔转账到支付宝/银行卡、C2C/B2C 现金红包、记账本等**资金类单据**状态变化后的通知（文档写明：不建议单靠这一篇完成整体接入）。 |
| **投递方式** | `POST`，`Content-Type: application/x-www-form-urlencoded`；公共参数含 **`notify_id`、`utc_timestamp`、`msg_method`、`app_id`、`version`、`biz_content`、`sign`、`sign_type`、`charset`** 等。 |
| **`msg_method`** | 固定为 **`alipay.fund.trans.order.changed`**。 |
| **`biz_content`** | JSON 字符串，内含 **`biz_scene`、`origin_interface`、`out_biz_no`、`product_code`、`order_id`、`status`、`action_type`、`trans_amount`** 等（与**商品支付订单**不是同一套字段）。 |
| **接收地址** | 文档中的 **`NOTIFY_URL`** 指在开放平台控制台为应用配置的 **应用网关地址**（消息服务投递入口），**不是**你在 `alipay.trade.page.pay` 里写的那个商品 **`notify_url`**。 |
| **应答** | 同步响应体需为字符串 **`success`**（成功）或 **`fail`**（失败会按策略重试；文档示例：约 25 小时内 8 次，间隔如 2m、10m、10m、1h…）。 |

**和本仓库的关系**：当前项目走的是 **`alipay.trade.page.pay` + 交易 `notify_url`**。若你没有接入「转账 / 红包 / 记账本」等资金接口，**一般不需要**处理 `alipay.fund.trans.order.changed`；在控制台订阅了这类消息，也不会替代网站支付的入账逻辑。

---

## 6. 小结：这样设计在解决什么问题？

1. **体验与安全**：收银台在支付宝完成，敏感操作不出支付宝域。  
2. **接入一致性**：网关统一入口，不同 `method` 扩展能力。  
3. **可靠入账**：异步通知 + 重试 + `success` 确认，避免只靠浏览器跳转。  
4. **防篡改**：请求侧商户签名、通知侧支付宝签名，双向可验证。  
5. **能力分域**：商品支付走交易 `notify_url`；资金转账/红包等走消息服务与应用网关，字段与入口都不同，需按各自文档接入。
